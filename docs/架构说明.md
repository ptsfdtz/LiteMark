# 架构说明

## 1. 架构目标

1. 按业务域分层，降低 `Layout` 单点复杂度。
2. 统一状态管理与错误处理约定。
3. 前端与 Rust 边界清晰，接口可演进可回滚。

## 2. 目标目录结构

```text
src/
  app/
    providers/            # i18n、theme、error boundary
    router/               # 若后续引入路由
    styles/               # 全局 token 与基础样式
  features/
    editor/               # Monaco 相关交互
    preview/              # Markdown 渲染与交互
    file-manager/         # 打开/保存/最近文件
    settings/             # 设置与偏好
  entities/
    document/             # 文档实体与模型
    preference/           # 主题/语言/工作目录等
  shared/
    ui/                   # shadcn/ui 组合与业务无关组件
    markdown/             # 渲染、安全、插件内核
    lib/                  # 通用工具（cn、path、guards）
    types/                # 共享类型
```

## 3. 状态管理规范

1. 组件内局部状态：仅用于 UI 瞬态（弹层开关、输入焦点）。
2. 业务状态：按 feature 建立 hooks/store，避免全部堆在 `Layout`。
3. 持久化状态：统一通过 `entities/preference` 访问，不直接散落 `localStorage`。

## 4. 接口层规范

前端调用 Rust 的约定：
1. 统一在 `shared/lib/tauri-client.ts` 封装 `invoke`。
2. 统一返回 `Result<T, AppError>` 风格，避免直接 `console.error`。
3. 命令命名保持动宾结构：`read_text_file`、`extract_toc`。

## 5. 错误处理规范

1. 禁止静默吞错（`catch {}`）除非显式注释原因。
2. 用户可见错误通过 toast/dialog 提示。
3. 开发错误记录统一日志层，避免在组件中散落 `console.*`。

## 6. 日志规范

日志分级：
1. `debug`：开发调试（默认关闭）。
2. `info`：关键流程。
3. `warn`：可恢复异常。
4. `error`：不可恢复或影响核心路径。

## 7. 命名规范

1. 组件文件：`PascalCase.tsx`
2. Hook：`useXxx.ts`
3. Store/Service：`xxx.store.ts`、`xxx.service.ts`
4. 类型：优先内聚在 feature 内，跨域类型放 `shared/types`。

## 8. 渐进迁移策略

1. 旧目录保留一段时间，使用桥接导出过渡。
2. 每次迁移一个 feature，不跨域大范围改动。
3. 每次 PR 必须同步更新对应文档。
