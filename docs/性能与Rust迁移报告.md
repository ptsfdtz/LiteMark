测试基线：

2. 当前无 Rust 性能基准输出。

## 2. 性能瓶颈初判

前端热点：
1. `Preview.tsx` 中 Markdown 渲染每次输入全量重算。
2. 代码高亮存在重复处理（rehype + 手动 highlight）。
3. `Layout.tsx` 状态集中，导致不必要重渲染。
4. Monaco 与预览未做渲染隔离（重计算传播范围大）。

## 3. Rust 下沉候选清单

| 候选模块 | 当前实现 | 预估收益 | 复杂度 | 优先级 |
| --- | --- | --- | --- | --- |
| TOC 提取 | TS 正则/AST（待实现） | 大文档目录生成加速 2~4x | 中 | P2-高 |
| 文档统计（字数/行数/段落） | TS 遍历（待实现） | 高频实时统计 CPU 降低 40%+ | 低 | P2-高 |
| 文档索引（工作目录） | Rust 已有 list_text_files 非递归 | 扩展后目录扫描延迟显著下降 | 中 | P2-中 |
| Markdown 预处理（中英文数学包装） | TS 正则替换 | 中大文档渲染耗时下降 20~35% | 中 | P2-中 |

## 4. 拟落地 Rust 样例

样例 A：`extract_toc`（Rust）
1. 输入：Markdown 文本。
2. 输出：标题层级树（level/text/slug）。
3. 前端用途：预览侧 TOC 面板、快速跳转。

样例 B：`calc_doc_stats`（Rust）
1. 输入：Markdown 文本。
2. 输出：字符数、单词数、行数、段落数、代码块数量。
3. 前端用途：状态栏实时统计。

## 5. 基准方案

方法：
1. 构造 3 组样本：50KB、500KB、1MB Markdown。
2. 每组执行 50 次，记录 p50/p95/max。
3. 对比 TS 与 Rust 两条实现路径。

指标：
1. 单次平均耗时（ms）。
2. p95 延迟（ms）。
3. 主线程阻塞时间（前端）。

目标阈值：
1. Rust 路径较 TS 路径平均耗时降低 >= 40%。
2. 大样本（1MB）p95 降低 >= 50%。

## 6. 迁移 trade-off

选择 Rust 的原因：
1. CPU 密集型文本分析在 Rust 更稳定，且可复用 Tauri invoke 通道。
2. 便于后续扩展到更大文档与目录索引。

放弃点：
1. 增加了跨语言调试复杂度。
2. 需要维护接口序列化与错误映射。

## 7. 回滚策略

1. 每个 Rust 能力都保留同名 TS fallback。
2. 通过 `rust_accel` 特性开关切换。
3. 若跨平台构建出现阻塞，短期回落到 TS 路径，不阻塞发布。
