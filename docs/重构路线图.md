# LiteMark 重构路线图

## 1. 目标与边界

本次重构目标是"可落地、可回滚、可验收"，采用渐进迁移，不做大爆炸重写。

核心目标（按优先级）：
1. 前端组件体系统一到 shadcn/ui。
2. Markdown 编辑/预览链路重构（正确性、安全性、可扩展性）。
3. 项目结构和工程文档重建，支持新人 30 分钟内启动。
4. 识别并下沉高频计算到 Rust，完成 1~2 个可测样例。

不在本轮范围：
1. 更换编辑器内核（Monaco 保留）。
2. 跨平台安装器 UI 设计重做。

## 2. 基线（2026-03-05）

工程现状基线：
1. `pnpm lint` 可通过，但存在 ESLint 配置迁移警告（`.eslintignore` 已废弃）。
2. `pnpm build` 可通过，但主包体积过大：`dist/assets/index-58wZj2g2.js = 2353.25 kB`（gzip 708.63 kB），已触发 Vite chunk 警告。
3. 自动化测试文件数量为 0（无单测/集成测试基线）。
4. 仓库缺少 `docs/` 结构化文档目录。

## 3. 分阶段里程碑（P0/P1/P2）

工期估算（单前端主程 + 1 名协作评审）：
1. P0：10 个工作日（2 周）
2. P1：15 个工作日（3 周）
3. P2：10 个工作日（2 周）
总计：35 个工作日（7 周）

### P0：工程基座与风险收敛（Week 1-2）

改动清单：
1. 引入 shadcn/ui 基础设施（`components/ui`、`utils/cn`、设计 token）。
2. 建立新目录骨架（`src/app`、`src/features`、`src/shared`、`src/entities`）。
3. 抽离 Markdown 渲染核心为独立模块（`shared/markdown`），实现安全默认策略。
4. 建立测试基础（Vitest + React Testing Library + Markdown 样例快照）。
5. 文档首轮补齐（README、CONTRIBUTING、架构说明、迁移说明、FAQ）。

风险点与回滚方案：
1. 风险：UI 基础样式变更导致视觉回归。
回滚：保留 `legacy-ui` 开关，支持按页面切回旧样式。
2. 风险：Markdown 安全策略过严导致历史内容显示异常。
回滚：引入 `safeMode` + 兼容白名单，按文件级开关回退。
3. 风险：目录迁移引发路径引用失败。
回滚：使用 `index.ts` 兼容导出层，逐步替换 import。

测试清单：
1. 单测：Markdown 语法用例（标题、列表、代码块、表格、引用、任务列表）至少 60 条。
2. 集成：编辑态/预览态/分屏态切换行为测试。
3. 手测：文件打开、保存、另存为、重命名、删除、主题切换、多语言。

验收标准（量化）：
1. 测试框架接入并在 CI 运行，P0 覆盖率（语句）>= 55%。
2. Markdown 核心语法回归通过率 >= 95%。
3. 文档可用：新成员按 README 在 30 分钟内完成首启（内部演练 2 人通过）。

### P1：功能迁移与一致性闭环（Week 3-5）

改动清单：
1. 业务组件全量迁移到 shadcn/ui 组合（详见《组件迁移清单》）。
2. Markdown 插件机制落地：目录（TOC）、高亮、数学公式可配置启停。
3. 编辑态/预览态/分屏态统一行为状态机（解决滚动同步和快捷键冲突）。
4. 安全策略落地：XSS 过滤、链接白名单、新开策略与 `rel` 约束。

风险点与回滚方案：
1. 风险：迁移期间组件样式闪烁或交互断裂。
回滚：按功能区拆分 PR，每个功能区可独立 revert。
2. 风险：插件扩展点设计不稳，后续插件接入成本高。
回滚：插件接口保持向后兼容，标记 `experimental` 并灰度发布。

测试清单：
1. 单测：组件行为与状态机测试。
2. 集成：Markdown 全链路（输入 -> AST -> 渲染 -> 安全过滤）。
3. 关键手测：大文件（1MB Markdown）编辑与预览稳定性。

验收标准（量化）：
1. UI 组件 100% 使用 shadcn/ui 或在清单中明确暂缓项。
2. Markdown 核心语法渲染正确率 >= 99%。
3. 高危 XSS 用例 0 漏洞（基于攻击样例集验证）。

### P2：性能与 Rust 下沉（Week 6-7）

改动清单：
1. 完成 1~2 个 Rust 下沉样例（优先 Markdown TOC 提取、文档统计/索引）。
2. 提供基准脚本和报告（优化前后对比）。
3. 前端包体优化（懒加载 Monaco/高亮资源、按需加载插件）。
4. 交付最终迁移说明和发布流程。

风险点与回滚方案：
1. 风险：Rust 接口变更导致前端调用失败。
回滚：保留纯 TS 实现作为 fallback，并通过 feature flag 切换。
2. 风险：跨平台构建复杂度上升。
回滚：在 CI 中保留纯前端路径，Rust 能力可选启用。

测试清单：
1. Rust 单测与基准测试。
2. 前后端联调测试（Tauri invoke 错误处理、超时处理）。
3. 性能回归：首屏、滚动同步、预览刷新耗时。

验收标准（量化）：
1. 首屏和关键交互路径性能提升 >= 20%。
2. 主包 JS 体积较基线减少 >= 25%。
3. Rust 样例任务吞吐提升 >= 2x 或 CPU 时间下降 >= 40%。

## 4. 资源评估

角色建议：
1. 重构负责人（1）：架构、代码合并、验收。
2. 前端开发（1）：组件迁移、状态与交互改造。
3. Rust 协作（0.5）：下沉模块与基准建设。
4. QA/评审（0.5）：回归测试和验收记录。

## 5. 发布与回滚策略

1. 分支策略：`refactor/p0-*`、`refactor/p1-*`、`refactor/p2-*`，按能力拆分。
2. 每阶段结束打一个可回滚标签：`refactor-p0-freeze`、`refactor-p1-freeze`。
3. feature flag：`ui_v2`、`markdown_pipeline_v2`、`rust_accel`。
4. 任一阶段验收失败可回滚到最近 freeze 标签。
