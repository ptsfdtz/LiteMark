# 开发规范（当前阶段）

> 目标：先把产品跑通，再逐步工程化，避免过早拆分。

## 1. 当前阶段原则

1. 业务先集中在 `App.tsx`，不强制拆分业务组件。
2. 只有基础复用能力放 `components/ui/*`。
3. 所有类型声明统一放 `src/types/*.d.ts`。
4. 组件命名保持简短，避免长名（如 `MdEditor`、`MdDoc`、`SaveBar`）。
5. 只有需要复用的业务块才封装成组件，不为“看起来整洁”而提前拆分。

## 2. 文件职责

1. `*.tsx`：仅 UI 结构与交互绑定。
2. `*.ts`：仅业务逻辑函数与工具函数。
3. `*.d.ts`：仅类型声明。

## 2.1 命名规范（组件）

1. 组件名建议 1~2 个单词，尽量不超过 16 个字符。
2. 组件名优先表达业务语义，不堆技术词。
3. 禁止出现“阶段性超长组件名”作为长期命名。

示例：
1. 推荐：`MdEditor`、`MdToolbar`、`SaveBar`
2. 不推荐：`InlineMarkdownEditorPanelContainer`

## 3. 目录约束（当前阶段）

```text
src/
  App.tsx
  main.tsx
  index.css
  components/
    ui/
  hooks/
  lib/
  types/
```

说明：
1. `components/common` 当前阶段不使用。
2. 不提前创建 `features/*` 业务拆分目录，等业务稳定后再迁移。

## 4. 组件封装规则（当前阶段）

满足任一条件才封装：
1. 同一 UI 片段在 2 个及以上位置复用。
2. 该片段存在独立交互状态且会持续演进。
3. 该片段拆出后能显著降低 `App.tsx` 复杂度。

不满足上述条件时，保持在 `App.tsx` 内实现。

## 5. 触发拆分条件（后续阶段）

满足任一条件再拆分：
1. `App.tsx` 超过 300~400 行且可读性明显下降。
2. 同一业务逻辑复用 >= 2 处。
3. UI 与逻辑耦合导致测试/维护成本显著上升。

## 6. 提交要求

1. 每次改动前后保持 `pnpm lint`、`pnpm build` 通过。
2. 涉及结构调整必须同步更新 `docs/`。
3. 优先小步提交，确保可回滚。
